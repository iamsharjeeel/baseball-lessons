- **Real bug found and fixed during verification, worth flagging explicitly**: GSAP defaults `immediateRender: false` for any tween carrying a `scrollTrigger`, so the initial `gsap.fromTo()` calls in `FadeUp`/`StaggerGroup` never actually applied their "from" (opacity:0) state before the trigger fired — content was silently visible the whole time with the reveal animation never playing. Fixed by switching to explicit `gsap.set()` (from-state) + `gsap.to()` (scrollTrigger'd to-state), which is also GSAP's own documented recommended pattern for this exact case. Caught by introspecting `ScrollTrigger.getAll()` and checking inline styles directly in a live page rather than trusting a `fullPage: true` Playwright screenshot, which turned out to be a misleading way to verify this — full-page screenshots don't perform a real scroll, so below-the-fold ScrollTrigger state at capture time doesn't reflect real user-scroll behavior either way; verification of scroll-gated motion has to happen via real `scrollIntoViewIfNeeded()` + computed-style checks on the actual animated element (not a descendant, which has its own independent, unaffected `opacity` computed style — cost me a round of debugging before I caught that too).
- **Second real bug found and fixed**: `useReducedMotion`'s initial `useState` read `matchMedia` synchronously in the lazy initializer, which runs during the client's first render — when the real OS/browser preference is `reduce`, that first client render disagreed with the server-rendered HTML (which always assumes `false`, since `window` doesn't exist during SSR), producing a React hydration-mismatch console error (#418) on every load for reduced-motion users even though the page visually self-healed. Fixed by always initializing to `false` and correcting in a `useEffect` (post-hydration), which is the standard pattern for this exact SSR/media-query conflict. Verified zero console errors afterward with reduced-motion emulated.
